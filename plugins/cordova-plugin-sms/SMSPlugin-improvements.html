#01
Cursor not null-checked or closed in finally block
listSMS() · deleteSMS()
Crash Risk ⌄

If ContentResolver.query() returns null (which it can on some devices or when permissions are denied), calling cur.moveToNext() immediately throws a NullPointerException. Even when it doesn't return null, an exception mid-loop leaves the cursor open — leaking resources. Wrapping in a null-check and try/finally fixes both.

listSMS() — before & after
−Cursor cur = ctx.getContentResolver().query(uri, null, "", null, null);
−while (cur.moveToNext()) { ... }
−cur.close();
+Cursor cur = ctx.getContentResolver().query(uri, null, "", null, null);
+if (cur == null) {
+ callbackContext.error("Failed to query SMS");
+ return null;
+}
+try {
+ while (cur.moveToNext()) { ... }
+} finally {
+ cur.close();
+}
#02
Deprecated createFromPdu — missing format parameter
createIncomingSMSReceiver()
Crash Risk ⌄

The single-argument form of SmsMessage.createFromPdu() was deprecated in API 23. On modern Android versions and with 3GPP2 carriers, it can return malformed or null messages. The format string is available directly from the intent bundle.

onReceive() in BroadcastReceiver
−SmsMessage sms = SmsMessage.createFromPdu((byte[]) pdus[i]);
+String format = bundle.getString("format");
+SmsMessage sms = SmsMessage.createFromPdu((byte[]) pdus[i], format);
#03

#04
getColumnIndex returns -1 silently
deleteSMS() · listSMS()
Warning ⌄

getColumnIndex() returns -1 when a column doesn't exist, causing cur.getInt(-1) to silently return 0 — producing wrong data with no error. getColumnIndexOrThrow() throws an IllegalArgumentException immediately, making the bug obvious and debuggable.

deleteSMS() · listSMS()
−int id = cur.getInt(cur.getColumnIndex("_id"));
−int read = cur.getInt(cur.getColumnIndex(READ));
+int id = cur.getInt(cur.getColumnIndexOrThrow("_id"));
+int read = cur.getInt(cur.getColumnIndexOrThrow(READ));
#05
Fragile SMS deduplication — same message silently dropped
onSMSArrive()
Warning ⌄

The lastFrom/lastContent guard is meant to prevent double-firing from the ContentObserver + BroadcastReceiver both triggering. However it's never reset, so if the same sender legitimately sends the same message twice, the second is silently discarded. A timestamp-based TTL check is far safer.

onSMSArrive()
−private String lastFrom = "";
−private String lastContent = "";
−if (from.equals(lastFrom) && content.equals(lastContent)) return;
−this.lastFrom = from; this.lastContent = content;
+private String lastFrom = "";
+private String lastContent = "";
+private long lastSmsTime = 0;
+private static final long DEDUP_WINDOW_MS = 2000; // 2 seconds
+long now = System.currentTimeMillis();
+if (from.equals(lastFrom) && content.equals(lastContent)
+ && (now - lastSmsTime) < DEDUP_WINDOW_MS) return;
+this.lastFrom = from; this.lastContent = content; this.lastSmsTime = now;

#07
StringBuffer → StringBuilder in md5()
md5()
Minor ⌄

StringBuffer is synchronized (thread-safe) and carries unnecessary overhead for local string building that will never be shared across threads. StringBuilder is the correct modern replacement for single-threaded string concatenation.

md5()
−StringBuffer hexString = new StringBuffer();
+StringBuilder hexString = new StringBuilder();
